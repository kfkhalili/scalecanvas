# Architecture, Security & Tests Audit — ScaleCanvas
**Date:** 2026-03-01  
**Auditor:** GitHub Copilot  
**Scope:** Full codebase review — architecture patterns, API security, DB/RLS, test coverage.

---

## 1. Architecture

### Strengths

| Aspect | Notes |
|---|---|
| **Stack coherence** | Next.js 16 App Router + Supabase SSR + AWS Bedrock + ReactFlow is well-matched to the product. |
| **BFF pattern** | All LLM invocations and token mutations happen server-side. No AI SDK key or AWS credential is ever exposed to the browser. |
| **Effect.ts for typed errors** | Consistent use of `Effect`, `Either`, `Option`, and `pipe` throughout services and API routes gives typed, exhaustive error handling with no `any`-typed catch blocks. |
| **Atomic RPCs** | `deduct_token_and_create_session` and `claim_trial_and_create_session` are `SECURITY DEFINER` Postgres RPCs that perform locking, deduction, and insertion in a single transaction, eliminating race conditions. |
| **Service/lib separation** | `services/` holds DB operations accepting a `ServerSupabaseClient`; `lib/` holds pure logic. Services are trivially injectable and unit-testable. |
| **Body size defence** | `MAX_CHAT_BODY_BYTES = 1_000_000` is checked at raw-text level before `JSON.parse`, capping Bedrock input budget. |

### Issues

**[ARCH-1] ✅ RESOLVED (2026-03-01) — `proxy.ts` implemented with SOTA pattern**

Next.js ≥ 16 renamed `middleware.ts` to `proxy.ts` (the file exports a named `proxy()` function, not `middleware()`; `export default` still works but the named export is canonical). The Supabase `@supabase/ssr` docs now recommend calling `supabase.auth.getClaims()` in the proxy rather than `getUser()`:

| Method | Where to use | What it does |
|---|---|---|
| `getClaims()` | proxy / middleware | Local JWT validation (signature + expiry). Triggers silent refresh. No GoTrue call. |
| `getUser()` | API routes, Server Components | Validates session with GoTrue. Authoritative but adds latency. |

**Two bugs were also fixed in the existing `proxy.ts`:**
1. Used `getUser()` (GoTrue round-trip on every request) — replaced with `getClaims()`.
2. `setAll` only wrote cookies to `response`, not to `request` first — meaning Server Components rendered in the same pass did not see the refreshed token. Fixed to follow the canonical Supabase double-write pattern.

Additionally, `lib/supabase/proxy.ts` was created as a standalone `updateSession` helper (matching the Supabase example repo structure) so the proxy entrypoint stays thin.

```ts
// proxy.ts (Next.js 16 entrypoint)
export async function proxy(request: NextRequest) {
  return await updateSession(request);  // delegates to lib/supabase/proxy.ts
}
```

---

**[ARCH-2] 🟠 MEDIUM — Duplicate `extractContent` helper**  
`extractContent` is copy-pasted identically in `app/api/chat/route.ts` and `app/api/sessions/[id]/conclusion/route.ts`. Should be extracted to `lib/chatHelpers.ts` (the file already exists) and imported from there.

---

**[ARCH-3] 🟠 MEDIUM — Long-lived IAM access keys**  
`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` are static long-lived credentials stored in env vars. For Vercel deployments, AWS supports OIDC federation via `aws-actions/configure-aws-credentials`; for ECS/Fargate, the execution role eliminates stored keys entirely. If keys are compromised they cannot be scoped down short of deletion.

---

**[ARCH-4] ℹ️ INFO — `SUPABASE_SECRET_KEY` declared but never consumed**  
`.env.example` lists `SUPABASE_SECRET_KEY`; README marks it optional. No code imports or uses it. Either remove it from the example to avoid confusion, or document its intended use case.

---

**[ARCH-5] ℹ️ INFO — `canvasEdge.test.ts` has no corresponding source module**  
`lib/canvasEdge.test.ts` tests `reactflow`'s built-in `addEdge` utility using fixtures; there is no `canvasEdge.ts` source file. The test still catches ReactFlow API regressions but the file name is misleading. Consider renaming to `canvasEdge.integration.test.ts` or inlining into `canvas.test.ts`.

## 2. Security

### Strengths

| Area | Notes |
|---|---|
| **Auth on every API route** | `supabase.auth.getUser()` (validates JWT against GoTrue, not just decodes it locally) guards every endpoint before any data access. |
| **RLS everywhere** | Every table has `enable row level security` with ownership-scoped policies. Default deny when no policy matches. |
| **`rate_limit_buckets` lockdown** | RLS enabled with zero policies → default deny for all roles. Table is accessible only through the `check_rate_limit` `SECURITY DEFINER` RPC. |
| **RPC security** | Both RPCs use `security definer` + `set search_path = ''` to prevent search-path injection. |
| **Stripe webhook verification** | `stripe.webhooks.constructEvent` verifies HMAC signature before any business logic. |
| **Prompt injection resistance** | Canvas labels are XML-escaped before prompt composition. A `terminate_interview` tool is wired to the LLM so override attempts trigger session termination. |
| **Session guardrails** | `getSessionIfWithinTimeLimit` checks existence, ownership, status, and time limit in one Effect chain before any LLM call. |
| **Security headers** | `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, HSTS with `includeSubDomains; preload`, `Referrer-Policy`, `Permissions-Policy` all set in `next.config.ts`. |
| **`simulate_expired` gate** | The test-bypass flag is rejected unless `NODE_ENV !== 'production'` or `ALLOW_SIMULATE_EXPIRED=true`, preventing production abuse. |

### Issues

**[SEC-1] 🔴 HIGH — `MAX_CONTENT_LENGTH` exported but never applied**  
`lib/api.schemas.ts` exports `MAX_CONTENT_LENGTH = 10_000` but never uses it in `MessageSchema`. The `content` field accepts an unbounded string. With `MAX_MESSAGES = 100`, a caller can send 100 × arbitrarily large strings, capped only by the 1 MB raw body limit rather than the intended per-message ceiling.

Fix in `lib/api.schemas.ts`:
```ts
const MessageSchema = z.object({
  role: z.enum(["user", "assistant", "system"]),
  content: z.union([
    z.string().max(MAX_CONTENT_LENGTH),
    z.array(
      z.object({ text: z.string().max(MAX_CONTENT_LENGTH) }).passthrough()
    ),
  ]),
});
```

---

**[SEC-2] 🔴 HIGH — No Content-Security-Policy header**  
The app renders LLM output via `react-markdown`. Without a CSP, any XSS vector in markdown output (e.g. a `javascript:` link or inline event attribute) has no browser-level containment. Add to `next.config.ts`:
```ts
{
  key: "Content-Security-Policy",
  value: [
    "default-src 'self'",
    "script-src 'self' 'unsafe-inline'",
    "style-src 'self' 'unsafe-inline'",
    "img-src 'self' data: https://unpkg.com",
    "connect-src 'self' https://*.supabase.co wss://*.supabase.co",
    "frame-ancestors 'none'",
  ].join("; "),
}
```
Note: `'unsafe-inline'` can be tightened further once nonces are wired through the Next.js CSP nonce mechanism.

---

**[SEC-3] 🟠 MEDIUM — No rate limiting on `/api/checkout`, `/api/sessions` POST, or `/api/auth/handoff`**  
Only the chat endpoint has a rate-limit gate. The checkout endpoint issues Stripe checkout sessions on every call; a tight loop can generate hundreds of dangling Stripe sessions. Apply the existing `checkRateLimit` pattern (already used in chat) with a lower ceiling, e.g.:
- `/api/checkout`: 10 req/min per user
- `/api/auth/handoff`: 5 req/min per user

---

**[SEC-4] 🟠 MEDIUM — Checkout `success_url` / `cancel_url` built from request `Origin` header**  
`app/api/checkout/route.ts`:
```ts
const origin = request.headers.get("origin") ?? process.env.NEXT_PUBLIC_SITE_URL ?? null;
```
`Origin` is user-controlled. If the app is ever behind a permissive reverse proxy, Stripe redirect URLs could be set to an attacker-controlled domain. Fix by making the env var primary:
```ts
const origin = process.env.NEXT_PUBLIC_SITE_URL ?? request.headers.get("origin");
```

---

**[SEC-5] 🟠 MEDIUM — No application-layer `user_id` filter in session mutation routes**  
`updateSession`, `deleteSession`, and `saveCanvasState` in `services/sessions.ts` issue queries without `.eq("user_id", userId)`. Ownership is enforced solely by RLS. This is correct for production Supabase but:
- A future call with a service-role or bypassed client would silently operate on any user's data.
- The `chat` and `conclusion` routes already do explicit ownership checks — session CRUD should be consistent.

Suggested defence-in-depth addition:
```ts
client.from("interview_sessions")
  .update(...)
  .eq("id", sessionId)
  .eq("user_id", userId)  // ← add
```

---

**[SEC-6] 🟡 LOW — `ALLOW_SIMULATE_EXPIRED` undocumented in `.env.example`**  
The guard logic is correct (`NODE_ENV !== 'production'`), but the flag is invisible in `.env.example`. A developer deploying to a staging cluster with `NODE_ENV=development` would unknowingly leave the bypass open. Add a commented entry:
```dotenv
# Set to "true" on preview/staging only to enable simulate_expired in conclusion endpoint.
# Never set in production.
# ALLOW_SIMULATE_EXPIRED=false
```

## 3. Tests

### Strengths

| Area | Notes |
|---|---|
| **Chat route coverage** | `app/api/chat/route.test.ts` covers 30+ cases: 401, body-too-large, invalid JSON, UUID validation, rate-limit 429, phase selection, session ownership 403, terminated session, time-expired session. |
| **Rate-limit unit tests** | `lib/rateLimit.test.ts` covers: allowed, denied, RPC error, and argument correctness. |
| **Guardrails unit tests** | `lib/chatGuardrails.test.ts` covers time-limit arithmetic, timer display, and `countdownEffectKey`. |
| **E2E JWT bypass guard** | `isLocalSupabase()` prevents hardcoded-secret JWT minting from running against hosted Supabase. |
| **Fixture isolation** | `lib/__fixtures__/canvas.ts` provides canonical shared test data. |
| **Transcript, canvas parser, prompts** | All tested with multiple edge cases. |

### Issues

**[TEST-1] 🔴 HIGH — No test for per-message content length limit**  
There is no test asserting a message with content > 10,000 chars is rejected with 400. The schema bug (SEC-1) is entirely invisible to the test suite. Both the schema fix and a test are needed:
```ts
it("returns 400 when message content exceeds MAX_CONTENT_LENGTH", async () => {
  mockedCreate.mockResolvedValue(fakeSupabaseClientAuthenticated());
  const req = buildAuthenticatedRequest({
    messages: [{ role: "user", content: "x".repeat(10_001) }],
    nodes: [], edges: [],
  });
  const res = await POST(req);
  expect(res.status).toBe(400);
});
```

---

**[TEST-2] 🟠 MEDIUM — No tests for `/api/checkout`**  
Zero coverage for: successful checkout session creation, missing/invalid `pack_id`, Stripe customer lookup failure, Stripe session creation failure, missing `NEXT_PUBLIC_SITE_URL`, and the `origin`-header fallback path.

---

**[TEST-3] 🟠 MEDIUM — No tests for the Stripe webhook handler**  
Key missing cases: valid `checkout.session.completed` event, invalid signature → 400, missing metadata → logs and returns 200, non-numeric `tokens` metadata, `creditTokensForPurchase` failure, and idempotency (same `session.id` twice).

---

**[TEST-4] 🟠 MEDIUM — No tests for canvas save/retrieve (`PUT /api/sessions/[id]/canvas`)**  
`CanvasBodySchema` validation, node/edge count limits, and ownership/RLS-based 403 are exercised only by E2E tests, which require a live Supabase instance.

---

**[TEST-5] 🟠 MEDIUM — `simulate_expired=true` production-mode rejection untested**  
The conclusion route test verifies that `simulate_expired` bypasses the time check. There is no complementary test asserting rejection when `NODE_ENV=production` and `ALLOW_SIMULATE_EXPIRED` is unset:
```ts
it("rejects simulate_expired=true in production without ALLOW_SIMULATE_EXPIRED", async () => {
  const originalEnv = process.env.NODE_ENV;
  Object.defineProperty(process.env, "NODE_ENV", { value: "production" });
  delete process.env.ALLOW_SIMULATE_EXPIRED;
  // ...assert 403 with TIME_NOT_EXPIRED_MESSAGE
  Object.defineProperty(process.env, "NODE_ENV", { value: originalEnv });
});
```

---

**[TEST-6] 🟡 LOW — `canvasEdge.test.ts` tests a `reactflow` built-in**  
See ARCH-5. The test still has regression value but should be clearly labeled and not counted toward project code coverage metrics.

---

**[TEST-7] 🟡 LOW — Authenticated cross-auth E2E suite skipped on hosted Supabase CI**  
`cross-auth-journeys.spec.ts` wraps suites in `test.skip` when `!isLocalSupabase()`. This means CI on hosted Supabase skips the majority of auth-flow tests. Consider a lightweight mock-auth strategy (e.g. Supabase test helpers or a dedicated CI Supabase project) so these tests run in all environments.
